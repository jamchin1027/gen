RTL Rule File for CHPI / DDIC Digital Helper Work
=================================================

Purpose
-------
This file records RTL mistakes found during the CHPI 413b/420b decoder review.
Before producing or revising RTL later, Codex should read this file first and
apply the checklist below.


Core Rules
----------

R001 - Do not present simulation-only RTL as production RTL.
  Mistake seen:
    Icarus PASS was described too close to "deliverable RTL".
  Required behavior:
    Every RTL output must be labeled as one of:
      - concept / trace code
      - simulation demo RTL
      - synthesis-intended RTL
      - production candidate RTL
  Verification requirement:
    If claiming synthesis-intended RTL, at minimum run syntax simulation and
    explicitly state that company lint/synthesis/timing sign-off is still
    required unless those tools were actually run.

R002 - Do not mix blocking and non-blocking assignment in one clocked process.
  Mistake seen:
    A posedge always block used blocking assignment for next-state variables
    and non-blocking assignment for flops in the same block.
  Required behavior:
    Clock/reset always blocks describe registers and must use non-blocking
    assignment for registered signals.
  Verification requirement:
    Search for "=" inside posedge/negedge state update blocks before delivery.
    Temporary combinational signals must not be assigned inside the DFF update
    block.

R003 - Match the team's readable RTL style; do not force q/n/w naming.
  Mistake seen:
    The first correction used a large next-state block with *_q, *_n, *_w
    naming. It was functionally clearer than the original, but it did not
    match this IC team's usual review style.
  Required behavior:
    Prefer meaningful signal names and split clock/reset always blocks by
    function or signal group:
      - raw input aligner
      - packet/header counter
      - label/restore state
      - RGB24 packer
      - RGB24 output
  Verification requirement:
    A reviewer should be able to draw the DFF groups directly from the always
    blocks without decoding a global q/n/w naming scheme.

R004 - Do not hide fixed datapath work inside a for-loop in final review RTL.
  Mistake seen:
    The area-lean parser used:
      for (k = 0; k < 3; k = k + 1)
    This is synthesizable when constant-bounded, but it reads like software and
    made the hardware structure less obvious.
  Required behavior:
    For this CHPI decoder, write the parser as explicit stage0 / stage1 /
    stage2 blocks. Each stage consumes at most one 7b group.
  Verification requirement:
    Final review RTL must not contain the parser loop or an unused k variable.

R005 - Avoid latch risk in combinational logic.
  Mistake seen:
    Earlier combinational style had many nested if branches and was hard to
    audit for complete assignment.
  Required behavior:
    Every combinational always block must assign all outputs at the top and
    every if branch must have an explicit else when it changes outputs.
  Verification requirement:
    Inspect each always @* block for complete defaults and explicit else paths.
    No combinational block may rely on an accidental retained value.

R006 - Make the hardware structure readable.
  Mistake seen:
    Input aligner, header/label FSM, restore datapath, and RGB packer were
    mixed together without clear hardware boundaries.
  Required behavior:
    The code must show the hardware stages in comments and naming:
      - input aligner
      - explicit 7b parser stage0 / stage1 / stage2
      - header/label state
      - restore datapath
      - RGB24 packer
      - optional debug observation path, if any
  Verification requirement:
    A digital engineer should be able to draw DFFs, muxes, counters, and
    datapath widths from the code without guessing.

R007 - Area is not simply "fewer operations per clock".
  Mistake seen:
    A one-group-per-clock decoder was considered, but continuous 20b input has
    no backpressure. That would need a large backlog buffer and may increase
    area.
  Required behavior:
    For fixed-rate input streams, preserve the required throughput first, then
    minimize buffers and remove non-production debug paths.
  Verification requirement:
    For CHPI[19:0], keep up with 20/7 groups per clock on average. The decoder
    therefore needs to consume up to three 7b groups per clock.

R008 - Avoid unnecessary debug datapath in production-intended RTL.
  Mistake seen:
    useful20 debug output had its own FIFO/counter datapath, increasing state
    and logic even though RGB24 is the real output.
  Required behavior:
    If a debug output is required by the interface but not production behavior,
    tie it off or isolate it behind an explicit parameter.
  Verification requirement:
    In the area-lean RTL, useful20_valid must be 1'b0 and useful20_data must be
    20'd0 unless the user explicitly asks to restore that debug path.

R009 - Keep buffer width justified by throughput and residue.
  Mistake seen:
    Buffer size was not explained clearly enough.
  Required behavior:
    State why each registered buffer and each temporary combinational bus
    exists.
  Earlier area-lean decoder lesson:
    raw_residue       : 6 registered bits, because after parsing up to three
                        7b groups the CHPI raw residue is only 0..6 bits.
    raw_bit_count     : 3 registered bits is enough for 0..6.
    append_raw_shift  : 26 combinational bits, because one clock may append
                        20b to 0..6 old residue bits.
    rgb_residue       : 23 registered bits, because RGB24 packer residue is
                        always less than 24 bits after optional output emit.
    rgb_bit_count     : 5 registered bits is enough for 0..23.
    pack_shift        : 30 combinational bits, because 23 old bits plus one
                        7b payload group can exist before emitting RGB24.

R010 - Header/label must be explicit state, not hidden temporary behavior.
  Mistake seen:
    label_reg and replace_en were blocking-assigned inside the clocked parser.
  Required behavior:
    The first 7b group of each 420b packet updates active_label and
    restore_enable. That header group must not enter the RGB data packer.
  Verification requirement:
    Packet counter reset point must be exact:
      count 0   : header {H, Label}
      count 7..413 : 59 payload groups
      count 413 payload group returns next count to 0

R011 - Output valid/data should have one clear generation path.
  Mistake seen:
    rgb_valid/rgb_data were assigned in several places inside the same clocked
    block.
  Required behavior:
    Generate RGB output from one output always block. If the internal packer is
    combinational, feed it with one clear pack_emit_valid / pack_emit_data
    result.
  Verification requirement:
    rgb_valid and rgb_data should be registered in one clock/reset block.

R012 - Use exact bit order and width comments.
  Mistake seen:
    Some explanations were unclear about LSB-first order and bus packing.
  Required behavior:
    For every encoder/decoder RTL:
      - state whether input is LSB-first or MSB-first
      - state input bus width
      - state output bus width
      - state packet/header widths
      - state residue/buffer strategy

R013 - Simulation must be run before updating user-facing HTML/site.
  Mistake seen:
    Documentation was polished before the RTL quality was sufficiently checked.
  Required behavior:
    For any HTML page that presents RTL as corrected, run the relevant testbench
    first and show:
      - simulator name
      - command
      - PASS/FAIL result
      - tested data scope

R014 - Minimize registered width before polishing RTL style.
  Mistake seen:
    A readable Ver.B implementation still kept the 26b raw aligner and 30b RGB
    packer as registered state. Functionally it passed, but it was not the
    smallest area style.
  Required behavior:
    For area-first stream RTL, only register the residue needed across clocks.
    Wide aligner/packer buses may exist as combinational temporary signals, but
    should not become flops unless timing closure requires pipelining.
  Verification requirement:
    Before delivering "area-lean" RTL, list the real flop state and check that:
      - CHPI raw residue is registered as 0..6 bits, not a full 26b aligner.
      - RGB residue is registered as 0..23 bits, not a full 30b packer.
      - Packet position uses the smallest natural counter; for CHPI 420b this
        is a 6b group counter for header + 59 payload groups.

R015 - Avoid generic variable shifters in small datapath glue.
  Mistake seen:
    Ver.B used variable shifts such as:
      chpi_in << raw_bit_count
      payload_bits << pack_count
    Synthesis may optimize them, but they can be read as barrel shifters and
    make area less predictable.
  Required behavior:
    If the shift amount has a small finite set, write explicit case placement
    or a case-based helper function.
  Verification requirement:
    Search final area-lean RTL for "<<" and ">>".
    Constant shifts are acceptable. Variable shifts in the main datapath need
    an explicit area reason.

R016 - Do not put K-code alignment responsibility inside this decoder.
  Mistake seen:
    A robust-start decoder version was described too close to "auto align".
    That was misleading. It can only wait for start/valid; it cannot find K,
    bit-slip, or packet phase by itself.
  Required behavior:
    Keep system partition explicit:
      CDR/S2P -> K-code detect/bit-slip/20b word align -> packet bit0 align
      -> CHPI 420b/413b long-code decoder -> RGB24 packer.
    The area-min O/min RTL assumes the upstream aligner releases this decoder
    with packet bit0 on the first active clock.
  Verification requirement:
    Any document or RTL comment must say whether packet start is already
    aligned. Do not claim "auto alignment" unless K detect and bit-slip logic
    are really implemented and tested.

R017 - For metal-limited small-height layout, optimize mux/routing first.
  Mistake seen:
    Some readable versions were functionally correct but still carried wide
    bit packers or extra state. That hurts a 3-metal, small-height layout
    because routing congestion can dominate.
  Required behavior:
    Prefer narrow residue state and byte-lane packing:
      - raw_residue[5:0]
      - subphase[2:0] + packet_phase[1:0]
      - byte_residue[6:0] and byte_bit_count[2:0]
      - rgb_byte_buf[15:0] and rgb_byte_count[1:0]
    Avoid restoring a 44b variable-shift packer unless timing/pipeline tradeoff
    proves it is better in the real library.
  Verification requirement:
    Compare versions by at least:
      - registered bits
      - generic mux/shift structure
      - estimated generic CMOS area
      - simulation PASS scope
      - stated alignment assumption


Current Verified RTL
--------------------

File:
  chpi_long_decoder_rgb24_area_min.v

Purpose:
  Production candidate, area-min CHPI 420b to 413b long-code decoder for
  continuous CHPI[19:0] input and RGB24 output.

Required system assumption:
  - Upstream K-code/bit-slip/word aligner has already aligned packet bit0.
  - The first active clock after release/reset is the first 20b word of a 420b
    packet.
  - No valid_i bubble is inserted inside the continuous stream.
  - This decoder does not perform K-code search or bit-slip.

Review-style choices in Ver.O/min:
  - Kept the team's readable split clock/reset always style by function:
      raw residue, phase counter, label state, byte/RGB packer, RGB output.
  - Kept combinational helper blocks small and fully assigned.
  - Removed useful20 debug FIFO/counter datapath by tying debug output off.
  - Preserved continuous 20b throughput by using fixed 20b-to-7b subphase
    placement.
  - Replaced the earlier 44b bit-level packer with:
      7b-to-byte residue packer + RGB24 byte-lane packer
  - Replaced the 5b 0..20 word counter with:
      packet_phase[1:0] + subphase[2:0]
  - Yosys generic CMOS estimate:
      B baseline  : 7534+
      I robust    : 5038+
      M byte pack : 4538+
      O/min       : 4406+

Simulation:
  Tool:
    Icarus Verilog
  Commands:
    iverilog -g2012 -Wall -o chpi_area_min_00_01_aligned_tb.out
      chpi_long_decoder_rgb24_tb_00_01_aligned.v
      chpi_long_decoder_rgb24_area_min.v
    vvp chpi_area_min_00_01_aligned_tb.out

    iverilog -g2012 -Wall -o chpi_area_min_aligned_tb.out
      chpi_long_decoder_rgb24_tb_aligned.v
      chpi_long_decoder_rgb24_area_min.v
    vvp chpi_area_min_aligned_tb.out

    iverilog -g2012 -Wall -o chpi_area_min_pattern_aligned_tb.out
      chpi_long_decoder_rgb24_pattern_tb_aligned.v
      chpi_long_decoder_rgb24_area_min.v
    vvp chpi_area_min_pattern_aligned_tb.out

    iverilog -g2012 -Wall -o chpi_area_min_line_aligned_tb.out
      chpi_long_decoder_rgb24_line_tb_aligned.v
      chpi_long_decoder_rgb24_area_min.v
    vvp chpi_area_min_line_aligned_tb.out
  Result:
    PASS: bytes 00 and 01 observed
    PASS: first 256 bytes matched 00..FF
    PASS: all pattern regressions matched
    PASS: complete line matched: 5 CTRL_L bytes + 960 RGB bytes
    Output RGB24 words=322, checked bytes=965

Limitations:
  - Icarus PASS is not company synthesis sign-off.
  - Company lint, synthesis compile, timing, reset policy, and CDC checks are
    still required before production handoff.


Pre-Delivery Checklist
----------------------

[ ] Did I label the RTL maturity level correctly?
[ ] Did I avoid blocking assignment inside DFF update logic?
[ ] Did I split registers by readable function/signal group?
[ ] Did I avoid forcing q/n/w naming if the team's style does not use it?
[ ] Did I avoid constant parser for-loops in final review RTL?
[ ] Are all combinational outputs fully assigned with explicit else paths?
[ ] Are hardware stages named and commented?
[ ] Are buffer widths justified?
[ ] Did I avoid debug logic unless explicitly needed?
[ ] Did I minimize real registered width before optimizing comments/style?
[ ] Did I avoid generic variable shifters in small placement logic?
[ ] Did I state whether K/packet alignment is upstream or inside this RTL?
[ ] Did I check the area/speed/routing indicators before changing style?
[ ] Did I verify throughput and backlog behavior?
[ ] Did I run the relevant testbench?
[ ] Did I paste the exact PASS log into the review document?
[ ] Did I avoid updating public HTML before simulation passed?
