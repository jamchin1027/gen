CHPI Verilog Fix Review

檔名：CHPI_verilog_fix_windows.html。這頁不是單純把註釋變漂亮，而是把這次 RTL 失誤、修正方式、小面積資料流、完整 area-lean RTL、Icarus 仿真 PASS、以及後續規則全部放在同一份文件。
深刻檢討

原本把可模擬 code 寫得太像正式交付 RTL，且 clocked block 裡混了 blocking / non-blocking。

小面積方向

保留 20b/clk 吞吐，但移除 debug datapath，使用 26b raw buffer + 30b RGB buffer。

驗證後更新

Icarus line testbench 已 PASS：5 CTRL_L bytes + 960 RGB bytes 全部比對成功。

這次錯在哪 review root cause

1. 可仿真不等於可交付

Icarus PASS 只能表示這個 testbench 下功能對，不能直接當成 synthesis sign-off。

2. blocking / non-blocking 混搭

原始 code 在 posedge block 裡用 = 做 next-state，又用 <= 做 DFF，reviewer 很難信任。

3. 電路分層不清

aligner、parser、label state、restore、packer 混在一起，工程師不容易直接畫電路。

4. 小面積思考不完整

真正的小面積要先守住吞吐，再縮 buffer/debug path；不是把單拍邏輯砍少就好。

小面積 RTL 主流程 20b CHPI -> RGB24

CHPI[19:0] input

每 clock 固定進 20bit，LSB first，沒有 backpressure。

26b raw aligner

append 20bit，最多 drain 3 組 7bit，raw buffer 不需要做成大 FIFO。

7b parser

每拍最多解析三組 7bit，守住 20/7 groups per clock 平均吞吐。

Header / Label state

每個 420b packet 的第一組 7bit 是 {H, Label}，只更新狀態。

Restore datapath

遇到 {Label,0}/{Label,1} 還原全 0 或全 1 的 7bit group。

30b RGB packer

最多 23bit residue + 7bit group，所以 30bit buffer 足夠。

RGB24 output

buffer >= 24bit 時輸出一個 RGB24 word，一拍最多一個。

怎麼改、為什麼
阻塞/非阻塞混搭

原本在 clocked block 裡用 = 算 next-state，又用 <= 更新 DFF。修改後組合邏輯和 DFF 分開，工程師看到 always @* 就知道那不是暫存器。

電路概念

*_q 是 DFF 現值，*_n 是下一拍，*_w 是組合中間線。這樣能直接畫出 register 與 mux/adder。

原始問題寫法

reg [5:0]  useful_cnt, useful_cnt_next;
...
useful_fifo_next = useful_fifo_next | (...);
useful_cnt_next  = useful_cnt_next + 6'd7;
修改後小面積 RTL
// useful20 is debug only, not production RGB path.
assign useful20_valid = 1'b0;
assign useful20_data  = 20'd0;
怎麼改、為什麼
小面積修正

原本為 debug output 多放一套 FIFO/counter/datapath。正式 RGB24 decoder 不需要這條路，直接 tie-off，少掉一批暫存器與 mux。

可讀性

把 useful20 標成 debug observation，避免 reviewer 以為它是主資料路徑的一部分。

原始問題寫法
// tempting but wrong for continuous input
// parse only one 7b group per clock
if (raw_cnt >= 7) begin
  pop_one_group();
end
修改後小面積 RTL
// continuous CHPI[19:0] input needs 20/7 groups/clk.
// Drain up to three 7b groups per clock to keep raw buffer small.
for (k = 0; k < 3; k = k + 1) begin
  if (raw_cnt_w >= 5'd7) begin
    group_in_w = raw_fifo_w[6:0];
    ...
  end
end
怎麼改、為什麼
面積不是只看單拍邏輯

一拍只解 7bit 看起來邏輯少，但 input 每拍固定進 20bit、沒有 backpressure，長時間會累積 backlog，需要更大的 FIFO，總面積反而可能更差。

吞吐需求

20bit/clk 連續進來，平均需要 20/7 組 7bit/clk 的解析率，所以一拍最多解 3 組是為了守住吞吐與小 buffer。

原始問題寫法
reg [31:0] rgb_fifo, rgb_fifo_next;
reg [5:0]  rgb_cnt, rgb_cnt_next;

reg [25:0] raw_fifo, raw_fifo_next;
reg [5:0]  raw_cnt, raw_cnt_next;
修改後小面積 RTL
reg [25:0] raw_fifo_q, raw_fifo_n;
reg [4:0]  raw_cnt_q, raw_cnt_n;

reg [29:0] rgb_fifo_q, rgb_fifo_n;
reg [4:0]  rgb_cnt_q, rgb_cnt_n;
怎麼改、為什麼
buffer 位寬收斂

raw buffer 保留 26bit：每拍加 20bit、最多 drain 21bit，計數上界低於 27。RGB buffer 保留 30bit：最多 23bit residue 再加一組 7bit。

可綜合意圖

counter 改成 5bit，位寬原因寫在註釋裡，讓 reviewer 知道不是隨便抓 32/6bit。

原始問題寫法

  label_reg  = group_in[5:0];
  replace_en = (group_in[5:0] != {6{~group_in[6]}});
end
修改後小面積 RTL

  label_w      = group_in_w[5:0];

replace_en_n = replace_en_w;
怎麼改、為什麼
blocking 放錯位置

原本 label_reg = ... 在 clock block 裡，像是暫存器又像組合線。修改後 header 更新先成為 label_n，下一拍進 label_q。

header 不進 RGB

420b packet 第一組 7bit 是 {H, Label}，只更新狀態，不送入 RGB packer。這個狀態邊界要寫清楚。

原始問題寫法

...

  rgb_data  <= rgb_fifo_next[23:0];
  rgb_valid <= 1'b1;

怎麼改、為什麼
單點 output

原本多處直接寫 rgb_valid/rgb_data，review 時很像隱性 priority mux。修改後只產生 rgb_valid_n/rgb_data_n，DFF block 統一 sample。

一拍一個 RGB24

rgb_emit_seen 明確限制一拍最多輸出一個 RGB24 word，輸出節奏可以被檢查。

小面積取捨說明 area-lean rationale
為何不是一拍只解 7bit

input 固定 20b/clk 連續進來，若只解 7b/clk，資料會累積，需要大 FIFO，面積不一定小。

raw buffer = 26bit

每拍 append 20bit，最多 drain 3 組 7bit，raw residue 上界低於 27bit。

RGB buffer = 30bit

RGB packer 最多 23bit residue，再加一個 7bit restored group，所以 30bit 足夠。

useful20 tie-off

debug observation path 不屬於正式 RGB 資料路徑，拿掉 FIFO/counter 可減少面積。

    .valid_i        (valid_i),
    .chpi_in        (chpi_in),
    .rgb_valid      (rgb_valid),

module chpi_420b413b_rgb24_stream_dec (
  input  wire        clk,
  input  wire        rst_n,
  input  wire        valid_i,
  input  wire [19:0] chpi_in,
  output reg         rgb_valid,
  output reg  [23:0] rgb_data,
  output wire        useful20_valid,
  output wire [19:0] useful20_data
);
  // useful20 is a debug observation path in earlier demos. Tie it off here to
  // keep the production RGB path smaller.
  assign useful20_valid = 1'b0;
  assign useful20_data  = 20'd0;

  // Small-buffer state for a continuous 20b input stream.
  // raw_cnt_q is bounded to 0..26 because each clock appends 20b and drains
  // up to three 7b groups. A slower one-group parser would require a large
  // backlog buffer when valid_i is continuous.
  reg [25:0] raw_fifo_q, raw_fifo_n;
  reg [4:0]  raw_cnt_q, raw_cnt_n;

  // RGB packer holds at most 30 valid bits: 23 old bits + one 7b group.
  reg [29:0] rgb_fifo_q, rgb_fifo_n;
  reg [4:0]  rgb_cnt_q, rgb_cnt_n;

  reg [8:0]  bit_cnt_420_q, bit_cnt_420_n;
  reg [5:0]  label_q, label_n;
  reg        replace_en_q, replace_en_n;

  reg        rgb_valid_n;
  reg [23:0] rgb_data_n;

  reg [25:0] raw_fifo_w;
  reg [4:0]  raw_cnt_w;
  reg [29:0] rgb_fifo_w;
  reg [4:0]  rgb_cnt_w;
  reg [8:0]  bit_cnt_420_w;
  reg [5:0]  label_w;
  reg        replace_en_w;
  reg        rgb_emit_seen;
  reg [6:0]  group_in_w;
  reg [6:0]  group_out_w;
  integer    k;

  always @* begin
    raw_fifo_w    = raw_fifo_q;
    raw_cnt_w     = raw_cnt_q;
    rgb_fifo_w    = rgb_fifo_q;
    rgb_cnt_w     = rgb_cnt_q;
    bit_cnt_420_w = bit_cnt_420_q;
    label_w       = label_q;
    replace_en_w  = replace_en_q;
    rgb_valid_n   = 1'b0;
    rgb_data_n    = rgb_data;
    rgb_emit_seen = 1'b0;
    group_in_w    = 7'd0;
    group_out_w   = 7'd0;

    // First drain an old RGB24 word if the packer already has enough bits.
    if (rgb_cnt_w >= 5'd24) begin
      rgb_data_n    = rgb_fifo_w[23:0];
      rgb_valid_n   = 1'b1;
      rgb_fifo_w    = rgb_fifo_w >> 24;
      rgb_cnt_w     = rgb_cnt_w - 5'd24;
      rgb_emit_seen = 1'b1;
    end

    // Append one CHPI[19:0] input word, LSB first.
    if (valid_i) begin
      raw_fifo_w = raw_fifo_w | ({{6{1'b0}}, chpi_in} << raw_cnt_w);
      raw_cnt_w  = raw_cnt_w + 5'd20;
    end

    // Throughput note:
    // Continuous 20b input needs 20/7 groups per clock on average, therefore
    // this parser must drain two or three 7b groups per clock. This keeps the
    // raw buffer small instead of storing a whole line or a large backlog.
    for (k = 0; k < 3; k = k + 1) begin
      if (raw_cnt_w >= 5'd7) begin
        group_in_w = raw_fifo_w[6:0];

        if (bit_cnt_420_w == 9'd0) begin
          // Header group: {H, Label}. It updates state and is not RGB data.
          label_w      = group_in_w[5:0];
          replace_en_w = (group_in_w[5:0] != {6{~group_in_w[6]}});
        end else begin
          // Payload group: restore the label-coded all-0/all-1 cases.
          if (replace_en_w && (group_in_w[6:1] == label_w)) begin
            group_out_w = group_in_w[0] ? 7'b1111111 : 7'b0000000;
          end else begin
            group_out_w = group_in_w;
          end

          rgb_fifo_w = rgb_fifo_w | ({{23{1'b0}}, group_out_w} << rgb_cnt_w);
          rgb_cnt_w  = rgb_cnt_w + 5'd7;

          if (!rgb_emit_seen && (rgb_cnt_w >= 5'd24)) begin
            rgb_data_n    = rgb_fifo_w[23:0];
            rgb_valid_n   = 1'b1;
            rgb_fifo_w    = rgb_fifo_w >> 24;
            rgb_cnt_w     = rgb_cnt_w - 5'd24;
            rgb_emit_seen = 1'b1;
          end
        end

        raw_fifo_w    = raw_fifo_w >> 7;
        raw_cnt_w     = raw_cnt_w - 5'd7;
        bit_cnt_420_w = (bit_cnt_420_w == 9'd413) ? 9'd0
                                                   : bit_cnt_420_w + 9'd7;
      end
    end

    raw_fifo_n    = raw_fifo_w;
    raw_cnt_n     = raw_cnt_w;
    rgb_fifo_n    = rgb_fifo_w;
    rgb_cnt_n     = rgb_cnt_w;
    bit_cnt_420_n = bit_cnt_420_w;
    label_n       = label_w;
    replace_en_n  = replace_en_w;
  end

  always @(posedge clk or negedge rst_n) begin
    if (!rst_n) begin
      raw_fifo_q    <= 26'd0;
      raw_cnt_q     <= 5'd0;
      rgb_fifo_q    <= 30'd0;
      rgb_cnt_q     <= 5'd0;
      bit_cnt_420_q <= 9'd0;
      label_q       <= 6'd0;
      replace_en_q  <= 1'b0;
      rgb_valid     <= 1'b0;
      rgb_data      <= 24'd0;
    end else begin
      raw_fifo_q    <= raw_fifo_n;
      raw_cnt_q     <= raw_cnt_n;
      rgb_fifo_q    <= rgb_fifo_n;
      rgb_cnt_q     <= rgb_cnt_n;
      bit_cnt_420_q <= bit_cnt_420_n;
      label_q       <= label_n;
      replace_en_q  <= replace_en_n;
      rgb_valid     <= rgb_valid_n;
      rgb_data      <= rgb_data_n;
    end
  end
endmodule

RTL Rule File for CHPI / DDIC Digital Helper Work
=================================================

Purpose
-------
This file records the RTL mistakes found during the CHPI 413b/420b decoder
review. Before producing or revising RTL later, Codex should read this file
first and apply the checklist below.


Core Rules
----------

R001 - Do not present simulation-only RTL as production RTL.
  Mistake seen:
    Icarus PASS was described too close to "deliverable RTL".
  Required behavior:
    Every RTL output must be labeled as one of:
      - concept / trace code
      - simulation demo RTL
      - synthesis-intended RTL
      - production candidate RTL
  Verification requirement:
    If claiming synthesis-intended RTL, at minimum run syntax simulation and
    explicitly state that company lint/synthesis/timing sign-off is still
    required unless those tools were actually run.

R002 - Do not mix blocking and non-blocking assignment in one clocked process.
  Mistake seen:
    A posedge always block used blocking assignment for next-state variables
    and non-blocking assignment for flops in the same block.
  Required behavior:
    Use a two-process style for non-trivial RTL:
      - always @* or always_comb: combinational next-state only, use "=".
      - always @(posedge clk or negedge rst_n): DFF update only, use "<=".
  Naming convention:
      *_q : registered current state
      *_n : next state sampled by DFF
      *_w : combinational work/intermediate value

R003 - Make the hardware structure readable.
  Mistake seen:
    Input aligner, header/label FSM, restore datapath, and RGB packer were
    mixed together without clear hardware boundaries.
  Required behavior:
    The code must show the hardware stages in comments and naming:
      - input aligner
      - 7b parser
      - header/label state
      - restore datapath
      - RGB24 packer
      - optional debug observation path, if any
  Review standard:
    A digital engineer should be able to draw DFFs, muxes, counters, and
    datapath widths from the code without guessing.

R004 - Area is not simply "fewer operations per clock".
  Mistake seen:
    A one-group-per-clock decoder was considered, but continuous 20b input has
    no backpressure. That would need a large backlog buffer and may increase
    area.
  Required behavior:
    For fixed-rate input streams, preserve the required throughput first, then
    minimize buffers and remove non-production debug paths.
  CHPI decoder conclusion:
    Continuous CHPI[19:0] input needs 20/7 payload-group parsing rate on
    average. To keep the raw buffer small, parse up to three 7b groups per
    clock. A slower parser needs a large FIFO or input stall mechanism.

R005 - Avoid unnecessary debug datapath in production-intended RTL.
  Mistake seen:
    useful20 debug output had its own FIFO/counter datapath, increasing state
    and logic even though RGB24 is the real output.
  Required behavior:
    If a debug output is required by the interface but not production behavior,
    tie it off or isolate it behind an explicit parameter.
  CHPI area-lean decision:
    useful20_valid = 1'b0 and useful20_data = 20'd0 in the area-lean RTL.

R006 - Keep buffer width justified by throughput and residue.
  Mistake seen:
    Buffer size was not explained clearly enough.
  Required behavior:
    State why each buffer width exists.
  CHPI area-lean decoder:
    raw_fifo_q  : 26 bits, because one clock appends 20b and parser drains up
                  to three 7b groups; raw_cnt is bounded below 27.
    rgb_fifo_q  : 30 bits, because packer can hold up to 23 old bits then add
                  one 7b group before emitting RGB24.
    raw_cnt_q   : 5 bits is enough for 0..26.
    rgb_cnt_q   : 5 bits is enough for 0..30.

R007 - Output valid/data should have one clear generation path.
  Mistake seen:
    rgb_valid/rgb_data were assigned in several places inside the same clocked
    block.
  Required behavior:
    Compute rgb_valid_n/rgb_data_n in combinational logic, then update the DFF
    once in the sequential block.
  If only one output word per clock is allowed:
    Use an explicit emit flag such as rgb_emit_seen to avoid multiple writes to
    the same output in one cycle.

R008 - Header/label must be explicit state, not hidden temporary behavior.
  Mistake seen:
    label_reg and replace_en were blocking-assigned inside the clocked parser.
  Required behavior:
    Use label_q/label_n and replace_en_q/replace_en_n. The first 7b group of
    each 420b packet updates header/label state and must not enter RGB data.

R009 - Use exact bit order and width comments.
  Mistake seen:
    Some explanations were unclear about LSB-first order and bus packing.
  Required behavior:
    For every encoder/decoder RTL:
      - state whether input is LSB-first or MSB-first
      - state input bus width
      - state output bus width
      - state packet/header widths
      - state residue/buffer strategy

R010 - Simulation must be run before updating user-facing HTML/site.
  Mistake seen:
    Documentation was polished before the RTL quality was sufficiently checked.
  Required behavior:
    For any HTML page that presents RTL as corrected, run the relevant testbench
    first and show:
      - simulator name
      - command
      - PASS/FAIL result
      - tested data scope


Current Verified RTL
--------------------

File:
  chpi_long_decoder_rgb24_area_lean.v

Purpose:
  Synthesis-intended, area-lean CHPI 420b to 413b decoder demo for continuous
  CHPI[19:0] input and RGB24 output.

Area-lean choices:
  - Remove useful20 debug FIFO/counter datapath by tying debug output off.
  - Keep continuous 20b throughput by parsing up to three 7b groups per clock.
  - Use 26b raw buffer and 30b RGB buffer instead of large line/backlog buffer.
  - Use two-process RTL style for readable combinational/DFF separation.

Simulation:

  Command:

  - Company lint, synthesis compile, timing, reset policy, and CDC checks are
